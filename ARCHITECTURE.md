# AI Testing Framework — Architecture

## Overview

A generic black-box testing framework for any AI agent. Sends predefined seed questions, drives multi-round follow-up conversations, and evaluates responses using multiple signals.

---

## Scoring Signals

| Signal | Method | Deterministic | LLM Call |
|---|---|---|---|
| Exact | Keyword matching on raw response | Yes | No |
| Semantic | Cosine similarity between embeddings | Yes | No (embedding API only) |
| Composite | Weighted combination of exact + semantic | Yes | No |
| Judge | LLM evaluates completeness + coherence | No | Yes |

---

## Per-Turn Analysis

| Turn | Exact | Semantic | Composite | Judge |
|---|---|---|---|---|
| Seed | ✓ | ✓ | ✓ | ✓ |
| Follow-up | ✗ | ✓ (if on-topic) | ✗ | ✓ |

---

## Embedding Strategy

**What gets embedded:**
- `SeedQuestion.expectedEmbedding` — expected answer, computed once on first upsert, reused forever
- `Conversation.responseEmbedding` — model's response, computed per turn and stored

**What is never embedded:**
- Seed questions (question text itself)

**Embedding model:** `gemini-embedding-001` (3072 dimensions)

---

## Follow-up Semantic Analysis Flow

Follow-up questions are dynamically generated and may not be directly related to the original seed question. The framework handles this as follows:

```
Follow-up question generated by questionnaire agent
        ↓
Embed follow-up question → followUpQuestionVec
        ↓
Find nearest seed question via pgvector cosine search (<=>)
comparing followUpQuestionVec against SeedQuestion.questionEmbedding
        ↓
Score < threshold (e.g. 0.5)?
        ↓ YES                                    ↓ NO
    [OFF TOPIC] — skip semantic          Use matched seed's expectedEmbedding
    Judge score only                     Compare against follow-up responseEmbedding
        ↓                                Compute cosine → semantic score ✓
    Why? Check two signals:
    - followUpVec ↔ seedQuestionVec → low → questionnaire drifted
    - followUpVec ↔ previousResponseVec → high → model led it off-topic
        ↓
    [OFF TOPIC - QUESTIONNAIRE] or [OFF TOPIC - MODEL]
```

**Key insight:** `[OFF TOPIC]` is the gate. The why-analysis (questionnaire vs model fault) only runs when off-topic is triggered and gives actionable signal on what to fix.

## LLM Call Budget (per full run)

Scenarios: General Inquiry (5 seeds × 2 follow-ups), Medicine (13 seeds × 2 follow-ups), Clinic Timings (7 seeds × 3 follow-ups)

| Turn | LLM calls | Embed calls |
|---|---|---|
| Seed (25 total) | 1 judge each = 25 | responseEmbedding + semantic(response+expected) = 3 each = 75 |
| Follow-up (57 total) | 1 questionnaire + 1 judge = 2 each = 114 | responseEmbedding + semantic(response) = 2 each = 114 |
| **Total** | **139** | **189** |

First run only: +25 embed calls for `expectedEmbedding` upsert → **214 embeds**

**Known inefficiency:** Seed response is embedded twice per turn (once for storage, once for semantic analysis). Fix: fetch stored `responseEmbedding` and `expectedEmbedding` from DB and compute cosine directly — saves ~50 embed calls per run.

### Embedding Call Reduction (after fix)

| Turn | Before | After | Saved |
|---|---|---|---|
| Seed (25) | 3 each = 75 | 1 each = 25 | 50 |
| Follow-up (57) | 2 each = 114 | 1 each = 57 | 57 |
| **Total** | **189** | **82** | **107 per run** |

```
Before:                          After:

Seed turn                        Seed turn
  embed(response)     ← stored     embed(response)     ← stored + returned
  embed(response)     ← wasted     fetch expectedVec   ← DB (free)
  embed(expectedAns)  ← wasted     cosine(r, e)        ✓
  cosine(r, e)        ✓

Follow-up turn                   Follow-up turn
  embed(response)     ← stored     embed(response)     ← stored + returned
  embed(response)     ← wasted     reuse expectedVec   ← already in memory
  cosine(r, e)        ✓            cosine(r, e)        ✓
```

## Adapter Pattern

The model under test connects via:
- **HTTP adapter** — `POST /chat`, `DELETE /chat/:id`, `GET /health`
- **Function adapter** — direct TypeScript callback (for programmatic use)

New adapters can be added by implementing the `ModelAdapter` interface.

                              ↓               ↓

## Insights Agent

Runs fully async after a test run completes, decoupled from the main flow via a Redis message queue.

```
Test Run Completes
       │
       ▼
index.ts → LPUSH insights:queue (report JSON)
       │
       └── returns immediately (non-blocking)

Redis Queue (insights:queue)
       │
       ▼
insights-worker.ts (separate process)
       │
       ├── BRPOP — waits for next report
       │
       └── InsightsAgent.analyze(report)
               │
               ├── For each scenario (parallel):
               │       ├── Rule-based pass
               │       │     model-rules.ts  → flag model issues
               │       │     framework-rules.ts → flag scenario/setup issues
               │       │
               │       └── LLM pass (parallel)
               │             model-owner-prompt.ts → deeper pattern analysis
               │             framework-prompt.ts   → ground truth + scenario gaps
               │
               └── Write output files
                     reports/analysis/{runId}-model-owner.json
                     reports/analysis/{runId}-framework.json
```

### Two Report Audiences

| Report | Audience | Focus |
|---|---|---|
| `model-owner.json` | Model/agent developer | Response quality, completeness failures, hallucinations, coherence, latency |
| `framework.json` | Test framework author | Ground truth gaps, scenario design, questionnaire quality, scoring anomalies |

### Issue Priority Levels

| Priority | Meaning |
|---|---|
| critical | Breaks core functionality — must fix before release |
| high | Significantly degrades experience |
| medium | Noticeable but not blocking |
| low | Minor, polish-level |

### Rule-based vs LLM Detection

- **Rules** detect deterministic signals: exact score = 0, completeness ≤ 0.5, latency > threshold, missing semantic scores, all seeds perfect
- **LLM** detects patterns across turns: consistent deflection of certain question types, subtle contradictions, scenario coverage gaps

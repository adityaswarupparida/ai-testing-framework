# AI Testing Framework ‚Äî Architecture

## Overview

A generic black-box testing framework for any AI agent. Sends predefined seed questions, drives multi-round follow-up conversations, and evaluates responses using multiple signals.

---

## Scoring Signals

| Signal | Method | Deterministic | LLM Call |
|---|---|---|---|
| Exact | Keyword matching on raw response | Yes | No |
| Semantic | Cosine similarity between embeddings | Yes | No (embedding API only) |
| Composite | Weighted combination of exact + semantic | Yes | No |
| Judge | LLM evaluates completeness + coherence | No | Yes |

## Embedding Strategy

**What gets embedded:**
- `SeedQuestion.expectedEmbedding` ‚Äî expected answer, computed once on first upsert, reused forever
- `Conversation.responseEmbedding` ‚Äî model's response, computed per turn and stored

**What is never embedded:**
- Seed questions (question text itself)

**Embedding model:** `gemini-embedding-001` (3072 dimensions)

---

## Follow-up Semantic Analysis Flow

Follow-up questions are dynamically generated and may not be directly related to the original seed question. The framework handles this as follows:

```
Follow-up question generated by questionnaire agent
        ‚Üì
Embed follow-up question ‚Üí followUpQuestionVec
        ‚Üì
Find nearest seed question via pgvector cosine search (<=>)
comparing followUpQuestionVec against SeedQuestion.questionEmbedding
        ‚Üì
Score < threshold (e.g. 0.5)?
        ‚Üì YES                                    ‚Üì NO
    [OFF TOPIC] ‚Äî skip semantic          Use matched seed's expectedEmbedding
    Judge score only                     Compare against follow-up responseEmbedding
        ‚Üì                                Compute cosine ‚Üí semantic score ‚úì
    Why? Check two signals:
    - followUpVec ‚Üî seedQuestionVec ‚Üí low ‚Üí questionnaire drifted
    - followUpVec ‚Üî previousResponseVec ‚Üí high ‚Üí model led it off-topic
        ‚Üì
    [OFF TOPIC - QUESTIONNAIRE] or [OFF TOPIC - MODEL]
```

**Key insight:** `[OFF TOPIC]` is the gate. The why-analysis (questionnaire vs model fault) only runs when off-topic is triggered and gives actionable signal on what to fix.

## LLM Call Budget (per full run)

Scenarios: General Inquiry (5 seeds √ó 2 follow-ups), Medicine (13 seeds √ó 2 follow-ups), Clinic Timings (7 seeds √ó 3 follow-ups)

| Turn | LLM calls | Embed calls |
|---|---|---|
| Seed (25 total) | 1 judge each = 25 | responseEmbedding + semantic(response+expected) = 3 each = 75 |
| Follow-up (57 total) | 1 questionnaire + 1 judge = 2 each = 114 | responseEmbedding + semantic(response) = 2 each = 114 |
| **Total** | **139** | **189** |

First run only: +25 embed calls for `expectedEmbedding` upsert ‚Üí **214 embeds**

**Known inefficiency:** Seed response is embedded twice per turn (once for storage, once for semantic analysis). Fix: fetch stored `responseEmbedding` and `expectedEmbedding` from DB and compute cosine directly ‚Äî saves ~50 embed calls per run.

### Embedding Call Reduction (after fix)

| Turn | Before | After | Saved |
|---|---|---|---|
| Seed (25) | 3 each = 75 | 1 each = 25 | 50 |
| Follow-up (57) | 2 each = 114 | 1 each = 57 | 57 |
| **Total** | **189** | **82** | **107 per run** |

```
Before:                          After:

Seed turn                        Seed turn
  embed(response)     ‚Üê stored     embed(response)     ‚Üê stored + returned
  embed(response)     ‚Üê wasted     fetch expectedVec   ‚Üê DB (free)
  embed(expectedAns)  ‚Üê wasted     cosine(r, e)        ‚úì
  cosine(r, e)        ‚úì

Follow-up turn                   Follow-up turn
  embed(response)     ‚Üê stored     embed(response)     ‚Üê stored + returned
  embed(response)     ‚Üê wasted     reuse expectedVec   ‚Üê already in memory
  cosine(r, e)        ‚úì            cosine(r, e)        ‚úì
```

## Adapter Pattern

The model under test connects via:
- **HTTP adapter** ‚Äî `POST /chat`, `DELETE /chat/:id`, `GET /health`
- **Function adapter** ‚Äî direct TypeScript callback (for programmatic use)

New adapters can be added by implementing the `ModelAdapter` interface.
